## 1. 输入栏未发送文字持久化

### 1.1 现状分析

| 层级 | 当前状态 |
|------|----------|
| **UI 状态** | `ChatViewModel._inputText: MutableStateFlow<String>` — 纯内存，无持久化 |
| **数据模型** | `Session.draft: String?` — 已定义但从未写入 |
| **数据库** | `SessionEntity.draft TEXT` — 列已存在，`SessionDao.updateDraft()` 方法已就绪 |
| **业务层** | `SessionManager.updateSessionDraft()` — 方法已就绪但从未被调用 |

**结论：基础设施完整，只差"连接管路"。**

### 1.2 根因

当前调用链存在三个断点：

```
用户打字 → updateInputText() → _inputText (内存)
                                        ↓ 断点①：未保存到 Session.draft
进入会话 → loadSession() → 从 DB 加载 session
                                        ↓ 断点②：未从 session.draft 恢复 _inputText
离开会话 → ChatScreen 销毁 → ViewModel.onCleared()
                                        ↓ 断点③：未触发 draft 保存
```

### 1.3 修复方案

**三处关键改动，修改量 < 50 行：**

#### 改动 A：进入会话时恢复草稿

文件：`ChatViewModel.kt` → `loadSession()`

```kotlin
fun loadSession(sessionId: String) {
    val existing = store.getSession(sessionId)
    if (existing != null) {
        _currentSessionId.update { sessionId }
        updateAgentName(existing.agentId)
        // ★ 新增：恢复草稿
        if (!existing.draft.isNullOrBlank()) {
            _inputText.update { existing.draft!! }
        }
        return
    }
    // ... 从 DB 加载后同样恢复 draft
}
```

#### 改动 B：会话切换 / 离开时保存草稿

文件：`ChatViewModel.kt` — 新增 `saveCurrentDraft()` 方法

```kotlin
fun saveCurrentDraft() {
    val sessionId = _currentSessionId.value ?: return
    val text = _inputText.value
    viewModelScope.launch {
        sessionManager.updateSessionDraft(sessionId, text.ifBlank { null })
    }
}
```

文件：`ChatScreen.kt` — 添加 `DisposableEffect` 在离开屏幕时保存

```kotlin
DisposableEffect(sessionId) {
    onDispose {
        chatViewModel.saveCurrentDraft()
    }
}
```

#### 改动 C：发送消息后清空草稿

文件：`ChatViewModel.kt` → `sendMessage()` — 已在第 206 行清空 `_inputText`，补充清空 DB draft：

```kotlin
fun sendMessage(text: String) {
    // ...
    _inputText.update { "" }
    viewModelScope.launch {  // 异步清空 DB draft
        sessionManager.updateSessionDraft(sessionId, null)
    }
    // ...
}
```

## 2. 后台生成

### 2.1 现状分析

当前生成架构：

```
ChatViewModel.generateMessage()
  └── generationJob = viewModelScope.launch {  ← 绑定 ViewModel 生命周期
        └── llmProvider.sendPrompt(request)
              └── flow.collect { chunk -> ... }
  }
```

**生命周期的致命问题：**

```
场景 A：用户按返回键离开会话
  → ChatScreen 从导航栈移除
  → ChatViewModel.onCleared() 被调用
  → generationJob?.cancel() + llmProvider.cancel()
  → ❌ 生成中断，已接收内容可能丢失（最后一次 DB flush 前的内容）

场景 B：用户按 Home 键切到后台
  → App 进入后台
  → Android 可能在 5~30 分钟后杀进程
  → ❌ 生成中断，内容丢失
```

### 2.2 根因

- `generationJob` 作用域是 `viewModelScope`，生命周期与 ChatScreen 绑定
- `ChatViewModel.onCleared()` 无条件取消生成
- 无 Foreground Service 保护，后台进程随时可能被杀
- 消息更新通过 `MessageManager.debouncedDbUpdate()` 写入 DB（500ms 防抖），但若进程被杀，未 flush 的内容会丢失

### 2.3 修复方案

#### 架构设计

```
                                ┌─────────────────────────┐
                                │   GenerationService      │
                                │   (Foreground Service)   │
                                │                         │
                                │  ┌───────────────────┐   │
          startGeneration()     │  │  SSE flow collect  │   │
  ChatViewModel ──────────────►│  │  ↓                │   │
                                │  │  ChatStore.update │   │
          stopGeneration()      │  │  ↓                │   │
  ChatViewModel ──────────────►│  │  Room DB write     │   │
                                │  │  ↓                │   │
                                │  │  Notification     │   │
                                │  └───────────────────┘   │
                                └─────────────────────────┘
                                          │
                              实时同步 ChatStore (app-scoped)
                                          │
                                          ▼
                                ┌─────────────────────────┐
                                │   ChatScreen (返回时)    │
                                │   observe ChatStore     │
                                │   → 自动显示最新消息     │
                                └─────────────────────────┘
```

#### 关键变更

##### 2.3.1 新增 `GenerationService`（Android Foreground Service）

- 持有自己的 `MessageManager` / `SessionManager`（读写 ChatStore + Room）
- 持有 `LlmProvider` 引用
- 显示 Notification："正在生成回复..."
- 通知栏操作：取消生成（PendingIntent → stopSelf）
- 使用 Application-scoped CoroutineScope (通过 NexaraApplication 提供)

##### 2.3.2 改造 `ChatViewModel`

```kotlin
fun sendMessage(text: String) {
    // ... 创建 USER + ASSISTANT 消息、写入 DB ...
    // ★ 改为启动 Service 而不是 viewModelScope.launch
    GenerationService.start(context, sessionId, assistantMsgId, userMsgId, text)
}
```

##### 2.3.3 消息同步策略

- SSE chunk → `ChatStore.updateMessageInSession()` → UI 实时更新
- `debouncedDbUpdate(500ms)` → Room 持久化
- 用户返回 ChatScreen 时：ChatStore 中已有最新消息，自动展示

##### 2.3.4 异常处理

| 异常场景 | 处理策略 |
|----------|----------|
| 网络断开 | SSE 连接超时 → Service 将消息标记为 ERROR → 通知"生成失败" |
| App 被杀 | Android 重启 Service（`START_STICKY`）→ Service 检测到无活跃生成，停止自己 |
| 生成完成 | Service 更新通知 → "回复已完成" → 5 秒后自动移除通知 |
| 用户手动取消 | 通知栏按钮 → PendingIntent → Service.stopGeneration() → stopSelf() |

##### 2.3.5 AndroidManifest 注册

```xml
<service
    android:name=".service.GenerationService"
    android:foregroundServiceType="dataSync"
    android:exported="false" />

<uses-permission android:name="android.permission.FOREGROUND_SERVICE" />
<uses-permission android:name="android.permission.FOREGROUND_SERVICE_DATA_SYNC" />
<uses-permission android:name="android.permission.POST_NOTIFICATIONS" />
```

## 3. 思考容器自动展开

### 3.1 现状分析

思考容器组件：`InlineThinkingRow`（`PipelineBubble.kt:262-366`）

```kotlin:262-272:PipelineBubble.kt
@Composable
private fun InlineThinkingRow(
    reasoning: String,
    isGenerating: Boolean,
    fontSize: Int
) {
    var isExpanded by remember { mutableStateOf(isGenerating) }
    LaunchedEffect(isGenerating) {
        if (isGenerating) isExpanded = true
    }
    // ...
}
```

调用方（`PipelineBubble.kt:120-129`）：

```kotlin
is PipelineStep.Thinking -> {
    val isLastThinking = allSteps.filterIsInstance<PipelineStep.Thinking>().lastOrNull() == step
    val isThinkingStreaming = isGenerating && isLastThinking && status == GenerationStatus.THINKING
    //                                                                    ^^^^^^^^^^^^^^^^^^^^^^^^
    //                                                                    ★ 问题所在
    InlineThinkingRow(
        reasoning = step.reasoning,
        isGenerating = isThinkingStreaming,
        fontSize = fontSize
    )
}
```

### 3.2 根因：时序竞态

生成流程的时序：

```
T0: generateMessage() 开始
T1: _generationStatus → THINKING
T2: flow.collect 开始，等待首个 chunk（TTFT：0.5~3 秒）
    → 此时 PipelineBubble 渲染：isThinkingStreaming = true （status == THINKING）
    → 但 step.reasoning 仍为空！（Message.reasoning 尚未更新）
    → buildPipelineSteps() 不创建 Thinking step
    → InlineThinkingRow 根本不存在于 UI 树中
────────────────────────────────────────────────
T3: 首个 chunk 到达 → _generationStatus → RECEIVING（行 344-346）
T4: chunk 含 reasoning 内容 → MessageManager.updateMessageContent(..., reasoning=...)
    → 100ms 节流后 ChatStore 更新
    → buildPipelineSteps() 现在创建 Thinking step
    → InlineThinkingRow 首次渲染，此时：
        isGenerating = isGenerating && isLastThinking && status == RECEIVING
                     = true      && true          && false
                     = false ← ❌ 不展开！
```

**核心矛盾：** 思考步骤的创建时机（Message.reasoning 非空）总是晚于 `status == THINKING` 的时间窗口。当思考步骤首次渲染时，status 已经是 RECEIVING，导致 `isThinkingStreaming = false`，容器永远不会自动展开。

### 3.3 修复方案

#### 原则

思考步骤的"正在生成"状态不应与 `GenerationStatus.THINKING` 绑定。应改为：**只要整体生成正在进行，且这是最后一个思考步骤，就一直保持"生成中"状态。**

#### 改动

文件：`PipelineBubble.kt` 第 123 行

```kotlin
// Before：
val isThinkingStreaming = isGenerating && isLastThinking && status == GenerationStatus.THINKING

// After：
val isThinkingStreaming = isGenerating && isLastThinking
```

**影响分析：**

| 场景 | Before | After | 正确？ |
|------|--------|-------|--------|
| TTFT 期（status=THINKING，无 reasoning） | 不渲染 Thinking step | 不渲染（reasoning 为空） | ✅ |
| 首个 reasoning chunk 到达（status=RECEIVING） | 渲染但 `isGenerating=false` → 折叠 ❌ | 渲染且 `isGenerating=true` → 展开 ✅ | ✅ |
| 推理完成，正文流式中（reasoning 不再增长） | 折叠，"思考完成" ✅ | "正在思考"（持续） ⚠️ | 见下文 |
| 生成结束 | 折叠，"思考完成" ✅ | 折叠，"思考完成" ✅ | ✅ |

**⚠️ 副作用：** 当推理完成但正文仍在流式输出时，思考行会持续显示"正在思考"的脉冲动画，而不是"思考完成"。

**评估：**
- Claude Desktop 等主流产品确实保持 thinking 展开直到整轮对话生成完毕
- 用户的原始需求是"自动打开折叠展示"，已满足
- 若后续需要更精细的"推理完成"判定，可引入 `Message.reasoningFinished: Boolean` 标志位

## 5. 风险与注意事项

### 5.1 后台生成的特殊风险

| 风险 | 等级 | 缓解措施 |
|------|------|----------|
| Foreground Service 耗电 | 中 | 仅 SSE 连接期间持有 WakeLock，完成后立即 stopSelf |
| 用户返回时 UI 状态不同步 | 中 | 利用 ChatStore (app-scoped StateFlow) 天然同步 |
| 通知权限被拒绝 | 低 | Foreground Service 在 API 34+ 必须显示通知，权限被拒则退化为普通后台任务 |
| 多会话同时生成 | 低 | `ChatState.currentGeneratingSessionId` 互斥，一个 Service 同一时间只服务一个会话 |
| DB 写入竞态 | 中 | MessageManager 已有 debounce 机制，Service 和 ViewModel 不应同时操作同一消息 |

### 5.2 思考展开的边界情况

- 历史已完成的思考消息不受影响（`isGenerating=false` → `isExpanded=false` → 折叠态）
- 如果用户手动折叠思考内容后，后续推理流式到来时不应覆盖用户选择 → 当前逻辑 `LaunchedEffect` 会覆盖，可增加 `userToggled` flag（可选优化，非必需）
