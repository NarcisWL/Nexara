# 会话界面Token统计功能审计报告

## 审计概述

**审计日期**: 2026-04-08  
**审计范围**: 会话界面Token统计指示器和统计面板  
**审计目标**: 评估当前实现的设计是否正确关注了会话上下文长度

---

## 一、当前实现概述

### 1.1 组件架构

- **ChatInputTopBar (Token指示器)**: 显示 `formatTokenCount(tokenUsage.total) TOK`，数据来源 `session.stats?.totalTokens`
- **StatsPanel (统计面板)**: 显示消息总数、Token使用、用户消息数、助手消息数，数据来源 `messages.reduce((sum, m) => sum + m.tokens.input + m.tokens.output)`

### 1.2 数据流分析

```
LLM API Response -> accumulatedUsage -> post-processor.ts updateStats -> session.stats.totalTokens -> ChatInputTopBar tokenUsage.total -> UI显示
```

**关键代码位置**:

1. Token指示器显示 - `ChatInputTopBar.tsx:126-131`
2. Token数据传递 - `app/chat/[id].tsx:650-653`
3. Token统计计算 - `post-processor.ts:186-204`
4. StatsPanel统计 - `StatsPanel.tsx:18-20`

## 二、发现的问题列表

### 🔴 严重问题 (P0)

#### 问题1: Token统计概念混淆 - 显示的是流量消耗而非上下文长度

**问题描述**:  
当前Token指示器显示的 `session.stats.totalTokens` 是**历史累计Token流量消耗**，而不是用户真正关心的**当前会话上下文长度**。

**用户期望**:
- 想知道：当前会话上下文用了多少token / 模型上下文窗口上限
- 目的：判断是否接近上下文限制，是否需要清理或新建会话

**当前实现**:
- 显示的是：所有消息的 input + output token 累计
- 含义：总共消耗了多少token流量

**影响范围**:
- 用户无法判断当前上下文是否即将溢出
- 用户可能误以为数字越大越危险，实际上该数字只增不减
- 误导用户做出错误的会话管理决策

---

#### 问题2: 缺少模型上下文窗口上限的获取和显示

**问题描述**:  
当前实现完全没有获取和显示模型的上下文窗口上限，无法让用户了解上下文使用比例。

**代码分析**:
- `modelConfig.contextLength` 存在于模型配置中（来源：`MODEL_SPECS` 或用户自定义）
- 但该数据没有传递给 `ChatInputTopBar` 组件
- `ChatInputTopBarProps` 接口中没有 `contextWindowLimit` 字段

---

### 🟠 中等问题 (P1)

#### 问题3: StatsPanel显示的Token统计含义不明确

**问题描述**:  
StatsPanel 显示的"Token使用"是消息中记录的token累计，与TopBar显示的 `session.stats.totalTokens` 来源不同，可能产生数据不一致。

---

#### 问题4: 没有实时估算当前上下文长度

**问题描述**:  
在用户输入时，没有实时估算即将发送的上下文总长度，无法帮助用户预判是否会超出限制。

---

### 🟡 轻微问题 (P2)

#### 问题5: Token格式化函数单位标识不直观

**问题描述**:  
当前显示格式为 `1.5K TOK`，"TOK"缩写不够直观，普通用户可能不理解。

---

#### 问题6: StatsPanel缺少上下文相关的统计信息

**问题描述**:  
StatsPanel 只显示了基础的统计信息，缺少对上下文管理有帮助的数据。

---

## 四、正确设计建议

### 4.1 核心设计原则

**会话界面Token统计应该关注的是"上下文长度"而非"流量消耗"**

用户在会话中的核心需求是：
1. 了解当前上下文使用了多少token
2. 了解模型的上下文窗口上限
3. 预判是否即将超出限制
4. 做出相应的会话管理决策

### 4.2 建议的UI设计

```
┌─────────────────────────────────────────┐
│  🤖 GPT-4o    📊 12.5K / 128K (10%)    │
│                              ████████░░░ │
└─────────────────────────────────────────┘
         ↑                ↑
      当前上下文      模型上限
```

### 4.3 建议的统计面板设计

```
┌─────────────────────────────────────────┐
│ 上下文使用情况                           │
├─────────────────────────────────────────┤
│ 当前上下文:     12,500 tokens           │
│ 模型上限:       128,000 tokens          │
│ 使用比例:       10% ████████░░░          │
├─────────────────────────────────────────┤
│ 上下文构成:                              │
│ • 会话历史:     8,200 tokens (65%)      │
│ • 系统提示词:   1,500 tokens (12%)      │
│ • RAG检索内容:  2,800 tokens (23%)      │
├─────────────────────────────────────────┤
│ 会话统计:                                │
│ • 消息总数:     15 条                   │
│ • 累计消耗:     25,000 tokens           │
└─────────────────────────────────────────┘
```

### 4.4 技术实现建议

1. **新增上下文长度计算Hook** - `useCurrentContextLength.ts`
2. **修改ChatInputTopBar组件** - 新增 `contextWindow` prop
3. **修改StatsPanel组件** - 增加上下文使用情况区域
4. **实时估算** - 在用户输入时显示预估上下文长度

---

## 五、结论

当前Token统计功能存在**核心设计缺陷**，将"流量消耗"误作为"上下文长度"展示给用户，导致用户无法正确判断会话状态。建议按照本报告的设计建议进行重构，重点关注
