## 详细组件分析

### Markdown解析器
- 功能概述：识别块级与行内LaTeX、SVG代码块，将Markdown切分为有序片段；若无特殊内容则返回整段文本。
- 关键点：
  - 使用正则收集匹配项并按起始位置排序，确保片段顺序正确。
  - 避免行内LaTeX与块级LaTeX相互覆盖，先处理块级再处理行内。
  - 末尾剩余文本作为普通文本段追加。
- 复杂度：O(n + k log k)，n为输入长度，k为匹配数量。

```mermaid
flowchart TD

Sort --> Build["遍历匹配构建片段序列"]
Build --> AppendText["追加剩余普通文本"]
AppendText --> EmptyCheck{"是否有片段？"}
EmptyCheck --> |否| OneText["返回整段文本"]
EmptyCheck --> |是| Done(["结束"])
```

### 文件处理工具
- 功能概述：读取文件为Base64，自动检测编码（UTF-8优先，其次jschardet检测），处理典型乱码特征，最终统一返回可读字符串；提供文件大小格式化。
- 安全与健壮性：
  - 优先UTF-8解码，若出现替换字符则进一步检测编码。
  - 对疑似"UTF-8被误读为GBK"的特征字符进行二次修正。
  - 统一异常捕获与错误消息包装。
- 性能建议：
  - 大文件建议分块读取或异步处理，避免阻塞主线程。
  - 复用Buffer与检测结果，减少重复分配。

```mermaid
flowchart TD
S(["开始"]) --> ReadB64["读取为Base64"]
ReadB64 --> ToBuf["转换为Buffer"]
ToBuf --> TryUTF8["尝试UTF-8解码"]
TryUTF8 --> Clean{"是否无替换字符？"}
Clean --> |是| RetUTF8["返回UTF-8内容"]
Clean --> |否| Detect["jschardet检测编码"]
Detect --> Decide{"是否高置信度GB2312/GB18030？"}
Decide --> |是| DecodeGBK["使用GBK解码"]
Decide --> |否| Fallback["兜底GBK解码"]
DecodeGBK --> AntiMojibake["检查乱码特征并强制UTF-8修正"]
Fallback --> AntiMojibake
AntiMojibake --> Ret["返回内容"]
TryUTF8 -.->|异常| Err["抛出错误"]
```

### 内容净化管线
- 功能概述：插件化净化流程，支持预保护（LaTeX规范化、AI文本清洗）、保护代码块、后保护（中英文间距、JSON修复、SVG校验）。

### 原生净化器（Sanitizer）
- **功能概述**：基于TurboModule的C++实现，提供5倍性能提升的热路径净化功能。
- **核心技术**：
  - 单次C++扫描：将标题修复、中英文间距、换行器三个插件合并为单一pass。
  - Unicode优化：使用原生Unicode处理，避免JS正则开销。
  - 线程隔离：在C++后台线程运行，零JS主线程阻塞。
- **API接口**：
  - `processHotPath(text, enableHeadingFix, enablePanguSpacing, enableLineBreak)`: 同步原生净化
  - `isNativeSanitizerAvailable()`: 检查原生模块可用性
- **性能指标**：相比JS实现提升约5倍速度，内存占用更低。

## 智能降级机制

### 降级策略架构
系统采用"原生优先，JS回退"的智能降级策略，确保在各种环境下都能正常运行：

```mermaid
flowchart TD
Start(["开始"]) --> CheckNative{"检查原生模块可用性"}
CheckNative --> |可用| TryNative["尝试原生实现"]
CheckNative --> |不可用| UseJS["使用JS实现"]
TryNative --> NativeSuccess{"原生调用成功？"}
NativeSuccess --> |是| ReturnNative["返回原生结果"]
NativeSuccess --> |否| FallbackJS["回退到JS实现"]
FallbackJS --> ReturnJS["返回JS结果"]
ReturnNative --> End(["结束"])
ReturnJS --> End
UseJS --> End
```

### 具体实现策略

#### 内容净化管线降级
- **热路径优化**：当原生净化器可用时，将标题修复、中英文间距、换行器合并为单次C++扫描。
- **部分降级**：原生失败时，自动回退到对应的JS插件组合。
- **阶段化降级**：post-protect和post-restore两个阶段分别进行降级。

#### 文本分词器降级
- **完全降级**：原生分词器不可用时，自动切换到JS Trigram实现。
- **估算降级**：原生估算失败时，使用JS估算算法。
- **性能保证**：JS实现仍提供良好的CJK文本处理能力。

#### 令牌计数降级
- **同步降级**：原生计数器不可用时，使用JS启发式算法。
- **精度权衡**：JS算法提供足够的精度，同时保持良好的性能。

## 性能考量
- UI线程让渡：批量处理使用小批次并发与短延时，避免主线程阻塞。
- WebView资源本地化：优先使用本地打包资源，失败时CDN降级，减少网络抖动与首屏等待。
- 图像缓存策略：生成缩略图后立即持久化，定期清理超期缓存，控制磁盘占用。
- 编码检测优化：优先UTF-8直解，仅在必要时进行复杂检测与二次修正。
- 尺寸缓存：LaTeX渲染组件对尺寸进行全局缓存，避免反复测量导致的布局抖动与重绘。
- **新增** 原生模块性能：原生净化器提供5倍性能提升，原生文本分词器消除JS主线程阻塞，原生令牌计数器避免正则开销。
- **新增** 智能降级：根据模块可用性动态选择最优实现，确保在所有环境下都有最佳性能。
