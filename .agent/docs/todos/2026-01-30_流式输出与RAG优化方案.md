# Nexara 流式输出与 RAG 性能审计报告与优化方案
**日期**: 2026-01-30  
**状态**: 待执行 (Pending)

## 1. 核心问题摘要

经过代码审计，确认 Nexara 在流式体验与响应速度上与 Cherry Studio / LobeHub 存在差距的主要原因如下：

| 问题现象 | 核心技术原因 | 严重程度 |
| :--- | :--- | :--- |
| **RAG 检索等待时间长** | **同步阻塞链**：LLM 请求被 `MemoryManager.retrieveContext` 强阻塞，且 RAG 内部包含串行的 `Query Rewrite` (大模型调用) -> `Embedding` -> `Rerank` 流程。 | 🔴 高 |
| **DeepSeek 模型流速慢/卡顿** | **过度节流 (Over-Throttling)**：UI 状态更新强制限制为 **200ms (5fps)**，导致本来 token 生成较慢的模型（如 DeepSeek R1）被进一步“切碎”和延迟。 | 🟠 中 |
| **Gemini Flash 瞬间吐出** | **节流副作用 + 缺乏插值**：Gemini Flash 生成极快（往往 <200ms），在 200ms 节流窗口内完成了大部分生成，导致 UI 仅收到 1-2 次大包更新。 | 🟠 中 |

---

## 2. 深度技术分析

### 2.1 RAG 延迟问题：同步阻塞架构
**定位文件**: `src/store/chat-store.ts`, `src/lib/rag/memory-manager.ts`

目前的调用链路是严格串行的：
```mermaid
graph LR
    User[用户发送] --> RAG_Start[RAG 检索开始]
    RAG_Start --await--> Rewrite[Query Rewrite (LLM调用)]
    Rewrite --await (最长15s)--> Embed[向量化]
    Embed --await--> Search[数据库搜索]
    Search --await--> Rerank[重排 (API调用)]
    Rerank --> LLM_Start[建立 LLM 连接]
    LLM_Start --> Stream[流式输出]
```

**关键瓶颈**:
1.  **Query Rewrite (查询重写)**: 这一步调用了大模型，带来了 **1-3秒** 的首字延迟（TTFT）。
2.  **Rerank (重排)**: 额外的 HTTP 请求，增加了延迟。
3.  **ChatStore 阻塞**: `generateMessage` 方法在检索完成前不会创建 LLM Client。

### 2.2 流式卡顿问题：200ms 节流阀
**定位文件**: `src/store/chat-store.ts` (Line 1169, 1325)

```typescript
const CONTENT_UPDATE_INTERVAL = 200; // 200ms (5fps)
```

**影响**:
*   **DeepSeek**: 积攒 4-5 个 token 才刷新一次，体感卡顿。
*   **Gemini Flash**: 200ms 内可能已经生成完一段话，导致瞬间显示。

## 3. 改进方案建议 (Action Items)

### 3.1 优化方案 A: RAG 响应速度提升 (Server-side)

- [ ] **移除/异步化 Query Rewrite**: 默认 **关闭** 同步的 Query Rewrite，仅在“深度检索”模式下开启。
- [ ] **并行预加载 (Speculative RAG)**: 在用户输入时后台预加载。
- [ ] **UI 反馈优化**: 明确展示“正在思考搜索词...”等细粒度状态。

### 3.2 优化方案 B: 流式体验平滑化 (Client/Store)

- [ ] **降低节流阈值**: 将 `CONTENT_UPDATE_INTERVAL` 从 **200ms** 降低至 **33ms** (30fps) 或 **50ms** (20fps)。
    *   Target: `src/store/chat-store.ts`
- [ ] **Typewriter Effect (可选)**: 针对 Gemini Flash 等极速模型，在 UI 组件层引入插值动画。

### 3.3 实施路线图

1.  **Step 1 (Immediate)**: 修改 `ChatStore.ts`，将 `CONTENT_UPDATE_INTERVAL` 降为 33ms。
2.  **Step 2**: 在 `ChatStore` 中优化 RAG 初始化逻辑，避免无效等待。
3.  **Step 3**: 全局配置优化，默认关闭 `enableQueryRewrite`。
